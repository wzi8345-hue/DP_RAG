# 本地离线记忆库 + 多用户多领域 设计方案

> 目标：在不联网的前提下，为 DP-RAG 增加**本地长期记忆库**，并支持**多用户、多领域**（如钢铁知识库 / 电池知识库）；切换领域时 system prompt、检索参数、词典、记忆命名空间随之整体切换；领域之间互不干扰。
>
> 设计原则：最大化复用现有成熟机制——三层配置 `_deep_merge`（`config.py`）、文件定义式 Skill（`routing/research_skills.py`）、按 collection 隔离的 KB（`api/routers/collections.py`）、`ClientRegistry` 连接池、`TaskStore` 异步任务、`SessionStore` 落盘。新增能力以**运行时 overlay + 命名空间隔离**的方式叠加，不改动基线行为。

---

## 0. 现状地基（方案依赖的既有能力）

| 能力 | 现状 | 复用方式 |
|------|------|----------|
| KB 数据隔离 | 每个知识库 = `kb_<slug>` Milvus 集合 + 本地工作目录；`collection` 按请求传入（`api/models.py` 的 `QueryRequest/ChatRequest`） | 领域绑定到 1+ 集合，数据天然隔离 |
| 文件定义式 Skill | `SKILL.md`(frontmatter) + plan/policy/synthesis；`skill_router` 触发词+LLM 置信度门控；坏档案降级跳过 | 领域档案 `DomainRegistry` **复刻**此模式 |
| 三层配置合并 | 默认 ← 文件 ← 运行时，`_deep_merge` 递归；`generation.system_prompt` 可文件加载 | 领域切换 = 叠加一层领域 overlay |
| 短期记忆 | `SessionStore`（TTL 30min，跨会话归零） | 长期记忆为全新模块，episodic 摘要在 `_cleanup` 钩子生成 |
| 用户身份 | 仅 `API_KEYS` 鉴权，无 `owner`；Milvus `owner` 字段已规划未启用 | **多用户硬前置**，需先打通 |
| 本地向量库 | 仓库已有 `milvus_lite.db` | 记忆库用 Milvus Lite，纯本地 |

**两个硬前置**：① 用户身份 `owner`；② 本地 embedding（主 embedding 在 `localhost:8002` 实为 SSH 隧道到远程 GPU，**不离线**，记忆库需本地小模型或 BM25 兜底）。

## 1. 核心概念模型：用户 × 领域 × 记忆

```
owner (用户)
  └── domain (领域: steel / battery / ...)
        ├── collections    →  绑定 1+ KB 集合 (kb_steel / kb_battery)
        ├── prompt profile →  领域专属 system_prompt + 综述结构
        ├── lexicon        →  领域术语·中英同义词典 (喂 router kw)
        ├── retrieval      →  领域调优检索参数 (权重/ef/阈值/类型配额)
        ├── skills         →  本领域启用的 Skill 子集
        └── memory ns      →  (owner, domain) 私有记忆命名空间
```

「领域」= 把 **数据 + 提示词 + 检索策略 + 记忆** 打包的档案。切换领域 = 切换整包。

### 隔离矩阵

| 维度 | 隔离手段 |
|------|----------|
| 文献数据 | 按 Milvus 集合（已具备） |
| 记忆数据 | `memory` 集合 + `partition_key = "{owner}:{domain}"`（pymilvus `partition_key_field`）；读写强制带 `(owner, domain)` 过滤 |
| 偏好(KV) | 本地文件路径 `memory/<owner>/<domain>.json` |
| 提示词/词典 | 领域档案目录 `domains/<id>/` |

---

## 2. 领域档案（Domain Profile）

文件定义式，对齐 Skill 的运营体验（用户目录可覆盖内置、热加载、坏档案降级）。

### 2.1 目录布局

```
domains/
  steel/
    DOMAIN.md          # frontmatter 元数据 + Description(分类说明)
    system_prompt.md   # 领域专家人设 + 回答规范
    lexicon.yaml       # 术语→英文同义词词典
    synthesis.md       # (可选) 领域综述结构
  battery/
    DOMAIN.md
    system_prompt.md
    lexicon.yaml
```

### 2.2 DOMAIN.md frontmatter schema

```yaml
id: steel                         # 唯一 id (=目录名)
name: 钢铁材料
description: 面向钢铁/合金的腐蚀、热处理、力学性能问答
collections: [kb_steel]           # 绑定的 Milvus 集合 (1+)
skills: [mechanism_analysis, comparison]   # 本领域启用的 skill 白名单 (空=全部)
retrieval:                        # 叠加在 config.retrieval 之上 (deep-merge)
  per_retriever_k: 24
  search: { dense: { ef: 192 } }
  progressive: { l2_per_path_k: 24 }
  # 可覆盖 quality_thresholds 的 by_route/by_type
memory:
  enabled: true
  top_k: 6                        # 注入的记忆条数上限
  min_confidence: 0.55            # 低于此置信度的记忆不注入
priority: 0                       # 自动匹配时的排序权重
triggers: ["钢", "合金", "steel", "alloy"]   # (可选) 自动推断领域用
```

### 2.3 lexicon.yaml（领域词典，驱动跨语言检索）

```yaml
# 术语 → 英文同义词/缩写/化学式 (注入 router, 自动补全 kw)
点蚀: [pitting corrosion, pitting]
回火: [tempering]
晶间腐蚀: [intergranular corrosion, IGC]
# 钢铁与电池各自独立, 互不污染
```

### 2.4 DomainRegistry（新增模块，复刻 load_skills）

- `load_domains(dirs)`：扫描目录、frontmatter 解析、`{id: DomainProfile}`、同 id 后加载覆盖。
- `resolve_domain(request)`：优先用请求显式 `domain`；否则由 `collection → domain` 反查；都没有则 `None`（回退基线行为）。
- 加载/解析失败降级跳过，不拖垮主链路（与 `research_skills` 一致）。

## 3. 领域切换时「系统随之优化」的机制

请求携带 `domain`（或由 `collection` 反查）。在 `flows/query.py` 查询入口解析领域档案后，做 5 处运行时 overlay（均叠加在现有链路上，不改基线）：

```
1. 生成提示词   system_prompt        ← domain.system_prompt   (覆盖 config.generation.system_prompt)
2. router 词典  router kw 自动补全    ← domain.lexicon         (强化"中文问句必带英文同义词"硬约束, 领域独立)
3. 检索参数     effective_retrieval  = _deep_merge(config.retrieval, domain.retrieval)
4. Skill 子集   skill_router 候选     ← domain.skills          (裁剪候选, 防跨领域误触发)
5. 记忆命名空间  memory 检索过滤       ← (owner, domain)
```

> 全部为**纯运行时 overlay**：不写回基线配置、不影响其它领域 → 即"互不干扰 + 切换即优化"。

## 4. 本地离线记忆库

### 4.1 存储选型

- **向量存储**：推荐 **Milvus Lite**（仓库已有 `milvus_lite.db`），嵌入式、无服务、纯本地；或复用现有 Milvus 单建 `memory` 集合。
- **隔离**：单 `memory` 集合 + `partition_key = "{owner}:{domain}"`，硬隔离 + 高效过滤检索，避免集合数膨胀。

### 4.2 离线 embedding（满足"不联网"）

- 主 embedding 走远程隧道，不能用于离线记忆。给记忆库配**本地小模型**（`bge-small-zh` / `Qwen3-Embedding-0.6B`，sentence-transformers 或 llama.cpp，CPU 可跑），**独立维度、独立集合**，与文献 embedding 解耦。
- **纯离线兜底**：无本地模型时退化为 **BM25/关键词记忆**（完全离线、零依赖、召回略弱）。
- 实现：`ClientRegistry` 增 `get_memory_embedder()`（本地 api_base/model），与文献 embedder 分缓存。

### 4.3 memory 集合 schema（建议）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | VARCHAR(主键) | 记忆条目 id |
| `owner` | VARCHAR | 用户 id（partition_key 组成） |
| `domain` | VARCHAR | 领域 id（partition_key 组成） |
| `mem_type` | VARCHAR | preference / fact / doc / episode / procedural |
| `text` | VARCHAR | 记忆正文（注入用） |
| `embedding` | FLOAT_VECTOR | 本地模型向量（独立维度） |
| `confidence` | FLOAT | 置信度（注入门控 + 覆盖判定） |
| `source_session` | VARCHAR | 来源会话 |
| `created_at` / `expires_at` | INT64 | 时间戳 / 过期（失效机制） |
| `ref_doc_ids` | VARCHAR(JSON) | 关联文献（可溯源） |

`partition_key_field = owner+domain` 拼接字段（或单独 `ns` 字段）。

### 4.4 偏好（结构化 KV，不向量化）

- 路径：`memory/<owner>/<domain>.json`，原子落盘（仿 `SessionStore` 的 `os.replace` tmp）。
- 内容：关注体系/方法/标准、术语习惯、答案语言粒度、默认 collection。
- 确定性读取，零向量开销。

## 5. 记忆内容分类（绑定 owner×domain）

| 类别 | 存储 | 内容 | 注入点 |
|------|------|------|--------|
| 偏好 | KV | 关注体系/方法、术语习惯、语言粒度、默认集合 | router kw + generation system |
| 事实/结论 | 向量 | 用户确认的结论、纠正的错误、自定义术语定义 | 检索式注入 context |
| 文献交互 | 向量+KV | 本领域常用/关注 `doc_id`（`doc_registry` 升级为长期） | 跨会话恢复文献锚点 |
| 情景摘要(episodic) | 向量 | 会话超 `max_turns` 后的滚动摘要（打 domain 标签） | 检索式注入 |
| 程序性(高级) | KV | 本领域 query→路由/kw 有效经验 | router few-shot |

## 6. 读写时机

### 写（提取）——异步，绝不阻塞回答
- 复用 `TaskStore` 后台线程：每轮结束抽取候选记忆 → 规则过滤 → **语义去重/合并** → 写入 `(owner, domain)`。
- 会话 TTL 过期前，在 `SessionStore._cleanup` 钩子生成 episodic 摘要落库。
- 支持用户显式"记住 X"。

### 读（注入）——三个集成点
1. **router 前**：偏好 + 领域词典改写 `kw`、定默认集合、定语言。
2. **检索阶段**：记忆作为一路 retriever（owner+domain 过滤）并入结果。
3. **generation 前**：画像 + 已确认结论注入 `system_prompt`。

> 纪律：**检索式注入 + 硬上限**（延续 `ChatSession` 的 token 膨胀防护），绝不无脑全量拼接。

## 8. 风险与铁律

- **记忆污染（头号风险）**：错误结论被反复注入会放大错误 → 记忆带 `confidence`、`expires_at`、可被新证据覆盖、**文献证据永远优先于记忆**、可溯源（`ref_doc_ids`）。
- **领域误判**：自动推断可能套错档案（钢铁问题用电池词典）→ **建议 UI 显式选领域/KB**，或用 `skill_router` 式高置信门控。
- **记忆膨胀 / token 预算**：写入去重+合并+重要性打分+TTL；注入设上限。
- **隔离正确性**：partition_key 过滤 + owner 鉴权双重保证；任何记忆读写必须带 `(owner, domain)`。
- **可控性/隐私**：用户能查看/删除自己的记忆。

## 9. 分阶段落地路线

| 阶段 | 交付 | 复用 | 前置 |
|------|------|------|------|
| **P0** | 领域档案 `DomainRegistry` + 切换即优化（system_prompt / lexicon / retrieval / skills overlay），**先不做记忆** | `config._deep_merge` + skill 加载模式 | 无 |
| **P1** | `owner` 用户身份打通（鉴权→owner + Milvus owner 字段） | 现有 API Key 体系 | — |
| **P2** | 本地离线记忆库（Milvus Lite + 本地 embedding / BM25 兜底）+ 异步提取 + 检索式注入，按 `(owner,domain)` partition 隔离 | `ClientRegistry`/`HybridRetriever`/`TaskStore` | P0+P1 |
| **P3** | 程序性记忆 + 记忆管理 UI | router few-shot + 前端 | P2 |

**P0 即可满足"切换领域 prompt/检索随之优化、互不干扰"**，且不依赖用户身份；真正的"长期记忆"从 P2 起，但务必先打通 `owner`。

---

## 10. 一句话总结

把「领域」做成**文件定义式档案 + 运行时 overlay**，把「记忆」做成 **`(owner, domain)` 命名空间下的本地向量库（Milvus Lite + 本地 embedding）+ KV 偏好**。前者复用三层配置合并与 Skill 模式，后者复用连接池与异步任务体系——多用户、多领域、互不干扰、切换即优化，且记忆库完全离线。落地从 P0（零前置、立即见效）起步，长期记忆待 `owner` 打通后进入 P2。
