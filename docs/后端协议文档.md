# DP-RAG 后端协议文档

> 科研文献 Agentic RAG 系统后端 API 协议。基于 FastAPI 实现，所有业务接口统一挂载在 `/api/v1` 前缀下。
>
> - **服务名**：DP-RAG Pipeline API
> - **版本**：1.0.0
> - **协议**：HTTP/1.1 + JSON；流式接口使用 SSE（Server-Sent Events）
> - **交互式文档**：服务启动后可访问 `/docs`（Swagger UI）、`/redoc`、`/openapi.json`

## 1. 通用约定

### 1.1 Base URL

```
http://<host>:<port>/api/v1
```

本地联调默认 `http://localhost:8080/api/v1`，前端开发服务器（Vite，端口 5173）通过 `/api` 代理到后端。

### 1.2 请求与响应格式

- 除文件上传外，所有请求体均为 `application/json`。
- 文件上传使用 `multipart/form-data`。
- 所有响应体（除 SSE 流）均为 `application/json`，使用 UTF-8 编码，中文不转义。

### 1.3 通用响应字段约定

- 时间戳字段（`created_at` / `updated_at` / `ts`）均为 Unix 秒级浮点时间。
- 集合名（`name`）以 `kb_` 为前缀，纯 ASCII；用户可见名为 `display_name`，可含中文。

### 1.4 错误响应

错误遵循 FastAPI 标准格式：

```json
{ "detail": "错误描述信息" }
```

| 状态码 | 含义 |
|--------|------|
| `200` | 成功 |
| `400` | 请求参数错误（如集合名非法、skills 未启用） |
| `401` | API Key 无效（仅在配置了 `API_KEYS` 时） |
| `404` | 资源不存在（任务、会话、skill 等） |
| `500` | 服务内部错误 |

> 注意：部分查询类接口（如日志详情）在资源不存在时返回 `200` + 空结构而非 `404`，详见各接口说明。

## 2. 认证

采用 **Bearer Token** 认证，由环境变量 `API_KEYS`（逗号分隔的多 key）控制：

- **未配置 `API_KEYS`**：所有接口免认证（本地联调默认）。
- **已配置 `API_KEYS`**：所有带认证的接口需在请求头携带：

```
Authorization: Bearer <your-api-key>
```

key 不匹配返回 `401 Invalid API key`。

> 运维类接口（`/stats`、`/health`、`/doc_summary`、日志接口）**不需要**认证。

## 3. 查询接口

### `POST /query`

单次查询：检索 → 生成，返回完整结果（非流式、无会话上下文）。

**认证**：需要

**请求体** `QueryRequest`：

| 字段 | 类型 | 必填 | 默认 | 说明 |
|------|------|------|------|------|
| `query` | string | 是 | — | 查询语句 |
| `mode` | string\|null | 否 | null | 检索模式：`hybrid` / `vector` / `metadata` |
| `top_k` | int\|null | 否 | null | 召回数量（null 用配置默认） |
| `use_agentic` | bool | 否 | true | 是否启用 Agentic RAG（LangGraph） |
| `professional` | bool | 否 | false | 专业研究模式：多轮递进式检索 + 综述综合 |
| `collection` | string\|null | 否 | null | 目标 Milvus 集合（null 用配置默认） |

**响应** `QueryResponse`：见 [QueryResponse / ChatResponse](#102-queryresponse--chatresponse)。

**示例**：

```bash
curl -X POST http://localhost:8080/api/v1/query \
  -H 'Content-Type: application/json' \
  -d '{"query":"耐候钢的耐腐蚀机理是什么？","professional":false,"top_k":3}'
```

## 4. 对话接口

### `POST /chat`

多轮对话：检索 + 生成，维护会话上下文，返回完整结果（非流式）。

**认证**：需要

**请求体** `ChatRequest`：

| 字段 | 类型 | 必填 | 默认 | 说明 |
|------|------|------|------|------|
| `query` | string | 是 | — | 查询语句 |
| `session_id` | string\|null | 否 | null | 会话 ID，null 时自动新建 |
| `use_agentic` | bool | 否 | true | 是否启用 Agentic RAG |
| `mode` | string\|null | 否 | null | 检索模式 |
| `top_k` | int\|null | 否 | null | 召回数量 |
| `stream` | bool | 否 | false | （本端点忽略此字段） |
| `professional` | bool | 否 | false | 专业研究模式 |
| `collection` | string\|null | 否 | null | 目标集合 |

**响应** `ChatResponse`：见 [QueryResponse / ChatResponse](#102-queryresponse--chatresponse)，额外包含 `session_id`。

---

### `POST /chat/stream`

SSE 流式对话。请求体同 `/chat`（忽略 `stream` 字段）。检索完成后逐块推送 LLM 输出。

**认证**：需要

**响应**：`Content-Type: text/event-stream`

- 响应头包含 `X-Session-Id`（本轮会话 ID）。
- 事件协议详见 [SSE 流式协议](#11-sse-流式协议)。
- `done` 事件中携带 `session_id`，前端据此维持多轮上下文。

**示例**：

```bash
curl -N -X POST http://localhost:8080/api/v1/chat/stream \
  -H 'Content-Type: application/json' \
  -d '{"query":"MoS2的晶格常数是多少？","professional":false}'
```

---

### `POST /sessions`

创建新会话，返回 `session_id`。

**认证**：需要

**响应**：

```json
{ "session_id": "239b0d9b422c4471" }
```

---

### `DELETE /sessions/{session_id}`

销毁指定会话。

**认证**：需要

**响应**：

```json
{ "deleted": true }
```

## 5. 灌入接口

灌入相关接口均为**异步任务**：提交后返回 `TaskResponse`（含 `id`），通过 `GET /tasks/{task_id}` 轮询进度。

### `POST /ingest/upload`

上传 PDF 并自动灌入指定知识库（一步完成：解析 → 分块 → 向量化 → 入库）。

**认证**：需要

**请求** `multipart/form-data`：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `collection` | string (Form) | 是 | 知识库名（支持中文，自动解析为 `kb_` slug） |
| `backend` | string (Form) | 否 | 解析后端：`mineru` / `uniparser` |
| `files` | file[] (File) | 是 | 一个或多个 PDF 文件 |

**行为说明**：
- 集合名经 `make_collection_slug` 解析；中文名生成 ASCII slug，原名存入工作目录元数据。
- 每篇 PDF 连同其中间产物落在 `<UPLOAD_DIR>/kb_<slug>/<原文件名>/` 下。
- 原始文件名（去后缀）作为文档目录名 / `doc_id`。
- 知识库不存在时自动创建。

**响应**：`TaskResponse`（`status="pending"`）。任务完成后 `result` 形如：

```json
{
  "collection": "kb_cfbd61d5",
  "files": 1,
  "success": 1,
  "failed": 0,
  "total_chunks": 35,
  "details": [
    {
      "doc_id": "钢铁热喷涂金属涂层耐蚀性能的研究进展",
      "total_chunks": 35,
      "steps": [
        {"step": "parse", "success": true, "elapsed": 7.01},
        {"step": "chunk", "success": true, "elapsed": 5.77},
        {"step": "embed", "success": true, "elapsed": 1.02},
        {"step": "store", "success": true, "elapsed": 1.26}
      ]
    }
  ]
}
```

---

### `POST /ingest/rebuild`

全量重灌（异步）：清空集合 → 从指定目录逐篇灌入。

**认证**：需要

**请求体** `IngestRequest`：

| 字段 | 类型 | 默认 | 说明 |
|------|------|------|------|
| `directory` | string | — | 解析产物根目录 |
| `recreate` | bool | false | （此端点强制 recreate） |
| `skip_existing` | bool | true | — |
| `backend` | string\|null | null | — |

**响应**：`TaskResponse`。

---

### `POST /ingest/append`

增量追加（异步）：不清空集合，同名 `doc_id` 覆盖。

**认证**：需要

**请求体** `IngestRequest`（同上）。`skip_existing=true` 时跳过集合中已存在的 `doc_id`。

**响应**：`TaskResponse`。

---

### `POST /ingest/parse`

仅解析 PDF（异步）：不做 chunk/embed/store。

**认证**：需要

**请求体** `ParseRequest`：

| 字段 | 类型 | 默认 | 说明 |
|------|------|------|------|
| `path` | string | — | PDF 文件路径或目录 |
| `output_dir` | string\|null | null | 中间产物输出目录 |
| `backend` | string\|null | null | `mineru` / `uniparser` |
| `timeout` | int | 1800 | 解析超时（秒） |

**响应**：`TaskResponse`。

---

### `POST /ingest/load-vec`

直接灌入已向量化的文件（异步）：跳过 parse/chunk/embed。

**认证**：需要

**请求体** `LoadVecRequest`：

| 字段 | 类型 | 默认 | 说明 |
|------|------|------|------|
| `path` | string | — | 目录 / glob / 单个 `.json` |
| `recreate` | bool | false | 灌入前是否清空集合 |
| `purge_existing` | bool | true | 是否按 `doc_id` 删除同名文档 |
| `skip_existing` | bool | false | 是否跳过已存在文档 |

**响应**：`TaskResponse`。任务完成后 `result` 形如 `{"files_loaded": N, "total_chunks": M}`。

---

### `GET /tasks/{task_id}`

查询异步任务状态。

**认证**：需要

**响应** `TaskResponse`：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | string | 任务 ID |
| `status` | string | `pending` / `running` / `done` / `failed` |
| `progress` | float | 进度 `0.0 ~ 1.0` |
| `result` | any\|null | 完成后的结果（结构因任务类型而异） |
| `error` | string\|null | 失败时的错误信息 |
| `created_at` | float | 创建时间戳 |

任务 ID 不存在返回 `404 Task not found`。已完成/失败的任务保留一段 TTL 后自动清理。

## 7. 知识库接口

每个知识库对应一个 Milvus 集合（`kb_` 前缀）+ 一个本地工作目录 `<UPLOAD_DIR>/kb_<slug>/`，后者收纳原始 PDF 与全部中间产物（解析结果 / 分块 / 向量化 JSON）。

### `GET /collections`

列出知识库：Milvus 已有集合 ∪ 本地工作目录（空库也可见）。

**认证**：不需要

**查询参数**：

| 参数 | 类型 | 默认 | 说明 |
|------|------|------|------|
| `prefix` | string | `kb_` | 集合名前缀过滤 |

**响应** `CollectionsListResponse`：

```json
{
  "collections": [
    { "name": "kb_a847ccfd", "display_name": "材料科学", "row_count": 117, "doc_count": 1 }
  ]
}
```

`CollectionInfo` 字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `name` | string | Milvus 集合名（`kb_` 前缀，纯 ASCII） |
| `display_name` | string | 用户可见名（可含中文），缺省回退为去前缀的 name |
| `row_count` | int | 集合中的数据块数（已 flush） |
| `doc_count` | int | 本地工作目录中已收纳的文档数（含尚未灌入的） |

---

### `POST /collections`

新建空知识库：创建本地工作目录，立即出现在列表中。Milvus 集合在首次上传灌入时自动创建。

**认证**：需要

**请求体** `CreateCollectionRequest`：

```json
{ "name": "材料科学" }
```

- 支持中文名：集合名生成 ASCII slug，中文显示名存入工作目录元数据 `.kb_meta.json`。
- 空名返回 `400 知识库名不能为空`。

**响应** `CollectionInfo`：

```json
{ "name": "kb_a847ccfd", "display_name": "材料科学", "row_count": 0, "doc_count": 0 }
```

---

### `DELETE /collections/{name}`

删除知识库：删除 Milvus 集合 + 连带清理本地工作目录（中间产物 / 原始 PDF）。

**认证**：需要

- `name` 必须以 `kb_` 开头，否则返回 `400`（默认集合 `literature_chunks` 不可通过此接口删除）。

**响应** `DeleteCollectionResponse`：

```json
{ "deleted": true, "name": "kb_a847ccfd" }
```

---

### `POST /collections/{name}/rebuild`

重建知识库（异步）：复用本地已存解析产物，清空集合后重新向量化入库（**不重新解析 PDF**，仅重跑 chunk → embed → store）。

**认证**：需要

- `name` 必须以 `kb_` 开头。
- 该知识库本地无解析产物时返回 `400`。

**响应** `TaskResponse`。任务完成后 `result` 形如：

```json
{ "collection": "kb_a847ccfd", "docs": 1, "success": 1, "failed": 0, "total_chunks": 117 }
```

## 8. 专家技能接口

专家技能（Skill）以文件定义，存放于内置目录（只读）与用户上传目录 `upload_dir`（可编辑）。专业研究模式（`professional=true`）下，技能路由器据用户查询自动选择最匹配的 skill。

### `GET /skills`

列出所有已加载的 skill（内置 + 用户），含可编辑的提示词正文。

**认证**：需要

**响应** `SkillListResponse`：

| 字段 | 类型 | 说明 |
|------|------|------|
| `enabled` | bool | skills 功能是否启用 |
| `router_mode` | string | 路由模式（`off` / ...） |
| `upload_dir` | string | 用户 skill 上传目录 |
| `skills` | SkillSummary[] | skill 列表 |

`SkillSummary` 关键字段：`id`、`name`、`description`、`priority`、`triggers`、`prefer_first_paths`、`sufficiency`、`tuning`、`guards`、`plan`、`policy`、`synthesis_system`、`synthesis_thinking`、`synthesis_user`、`editable`（是否可编辑，内置 skill 为 false）。

---

### `GET /skills/template`

返回新建 skill 的填写模版（字段说明 + 示例）。

**认证**：需要

**响应**：模版字典（字段含义 + 示例）。

---

### `POST /skills`

新建或覆盖一个用户 skill（写入 `upload_dir`），并触发研究 agent 重载（即时生效）。

**认证**：需要

**请求体** `SkillSpec`：

| 字段 | 类型 | 默认 | 说明 |
|------|------|------|------|
| `id` | string | — | skill 唯一标识 |
| `name` | string | — | 显示名 |
| `description` | string | "" | 描述 |
| `priority` | int\|null | 50 | 优先级（越大越先匹配） |
| `triggers` | string[] | [] | 触发关键词/条件 |
| `prefer_first_paths` | string[] | [] | 优先检索路径 |
| `sufficiency` | object | {} | 充分性判定参数 |
| `tuning` | object | {} | 调参参数（如 max_rounds 等） |
| `guards` | string[] | [] | 守卫规则 |
| `plan` | string | "" | 规划阶段提示词 |
| `policy` | string | "" | 策略阶段提示词 |
| `synthesis_system` | string | "" | 综述 system 提示词 |
| `synthesis_thinking` | string | "" | 综述思考提示词 |
| `synthesis_user` | string | "" | 综述 user 提示词 |

- skills 功能未启用时返回 `400`。
- 内容非法返回 `400`，写入/解析失败返回 `500`。

**响应** `SkillSaveResponse`：

```json
{ "saved": true, "id": "comparison", "skill": { /* SkillSummary */ } }
```

---

### `DELETE /skills/{skill_id}`

删除一个用户 skill（仅限 `upload_dir`；内置 skill 不可删），并触发重载。

**认证**：需要

- 未找到可删除的用户 skill 返回 `404`。

**响应** `SkillDeleteResponse`：

```json
{ "deleted": true, "id": "my_custom_skill" }
```

| `reranker` | string | `configured` / `disabled` / ... |
| `reflection` | string | `configured` / `disabled` / `inherits_generation` / ... |

### `GET /doc_summary`

按 `doc_id` 返回该文献的简介（summary 块），供前端引用角标点击展示。优先取 `type=summary` 块，缺失时回退到 `title` / 首个 `text` 块。

**认证**：不需要

**查询参数**：`doc_id`（string，必填）。

**响应** `DocSummaryResponse`：

| 字段 | 类型 | 说明 |
|------|------|------|
| `doc_id` | string | 文献 ID |
| `doc_name` | string | 文献名 |
| `title` | string | 标题 |
| `year` | int\|null | 出版年份 |
| `summary` | string | 简介正文 |
| `found` | bool | 是否找到 |

---

### `GET /logs/sessions`

返回有日志的 session 列表，按最后更新时间倒序。

> 日志按 `session_id` 收集自 pipeline 检索流程（`pipeline.*` logger 的 INFO 级别），**仅内存存储**，服务重启后清空。

**认证**：不需要

**响应** `LogSessionListResponse`：

```json
{
  "sessions": [
    {
      "session_id": "53ecbce07a794e57",
      "query": "锌铝镁镀层的主要优势有哪些？",
      "created_at": 1780885832.34,
      "updated_at": 1780885863.53,
      "line_count": 92
    }
  ]
}
```

---

### `GET /logs/sessions/{session_id}`

返回指定 session 的检索流程日志。

**认证**：不需要

**查询参数**：

| 参数 | 类型 | 说明 |
|------|------|------|
| `tail` | int\|null | 仅返回最后 N 行 |

**响应** `LogSessionDetail`：

```json
{
  "session_id": "53ecbce07a794e57",
  "query": "锌铝镁镀层的主要优势有哪些？",
  "created_at": 1780885832.34,
  "updated_at": 1780885863.53,
  "line_count": 92,
  "lines": [
    {
      "ts": 1780885837.12,
      "timestamp": "10:30:37",
      "level": "INFO",
      "logger": "pipeline.retrieval.langgraph_agent",
      "message": "[f6ead234] [run] final_phase='context_build' ..."
    }
  ]
}
```

> session 不存在时返回 `200` + 空 `lines` 数组（非 `404`）。

---

### `GET /logs/sessions/{session_id}/stream`

SSE 实时推送指定 session 的新日志行。

**认证**：不需要

**响应**：`Content-Type: text/event-stream`

- 每条新日志行作为一个 `data:` 事件推送，payload 为 `LogLineEntry` 结构。
- 每 15 秒发送一条 SSE 注释行 `: keepalive` 作为心跳，防止代理超时断开。

**事件示例**：

```
data: {"ts":1780885837.12,"timestamp":"10:30:37","level":"INFO","logger":"pipeline.flows.query","message":"..."}

: keepalive

```

## 10. 核心数据结构

### 10.1 Hit（检索命中片段）

`hits` 数组中的单条命中（字段视检索路径而定，可能部分缺省）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `pk` / `chunk_id` | string | 块主键 / 块 ID |
| `doc_id` | string | 文献 ID |
| `doc_name` | string | 文献名 |
| `type` | string | 块类型（title / summary / text / ...） |
| `section` | string | 所属章节 |
| `page_start` | int | 起始页 |
| `paragraph_index` | int | 段落序号 |
| `publication_year` | int | 出版年份 |
| `content` | string | 块正文 |
| `context` | string | 上下文 |
| `related_assets` | object[] | 关联资源（图表等） |
| `score` | float | 相关性分数 |
| `rrf_score` | float | RRF 融合分数 |
| `rerank_score` | float\|null | 重排分数 |
| `sources` | string[] | 召回来源（vector / metadata / ...） |
| `matched_keywords` | string[] | 命中关键词 |
| `subquery_id` | string | 所属子查询 ID（专家模式） |

### 10.2 QueryResponse / ChatResponse

两者结构基本一致，`ChatResponse` 额外含 `session_id`：

| 字段 | 类型 | 说明 |
|------|------|------|
| `query` | string | 原始查询 |
| `answer` | string | 生成的回答 |
| `hits` | Hit[] | 检索命中片段 |
| `context` | string | 拼装的上下文 |
| `usage` | object\|null | token 用量（`prompt_tokens` / `completion_tokens` / `total_tokens`） |
| `latency_s` | float | 端到端耗时（秒） |
| `session_id` | string | 会话 ID（仅 ChatResponse） |
| `error` | string\|null | 错误信息 |
| `needs_clarify` | bool | 是否需要澄清 |
| `needs_reuse` | bool | 是否复用上轮结果 |
| `no_answer` | bool | 是否无法回答 |
| `retry_count` | int | 重试次数 |
| `correlation_id` | string | 关联 ID（用于追踪日志） |
| `research` | object\|null | 专业研究模式执行概要（见下），普通模式为 null |

### 10.3 ResearchMeta（专业研究模式概要）

仅在 `professional=true` 时填充：

| 字段 | 类型 | 说明 |
|------|------|------|
| `status` | string | `complete` / `insufficient` / `clarify` / `error` |
| `rounds` | int | 检索轮数 |
| `evidence_docs` | int | 证据文献数 |
| `evidence_chunks` | int | 证据块数 |
| `gaps` | string[] | 证据缺口列表 |
| `covered` | string[] | 已覆盖的子问题 |

## 11. SSE 流式协议

`POST /chat/stream` 的事件流。每个事件为一行 `data: <JSON>\n\n`。`type` 字段区分事件种类：

### `status` — 阶段状态

```json
{ "type": "status", "stage": "retrieving" }
```

`stage` 取值：`retrieving`（检索中）/ `generating`（生成中）。

### `thinking` — 思考过程（专家模式）

```json
{ "type": "thinking", "content": "...", "round": 1, "phase": "synthesis" }
```

- 规划 / 每轮评估为整段下发；综述阶段（`phase="synthesis"`）为 token 流。
- `round`、`phase` 为可选字段。

### `text` — 正文 token

```json
{ "type": "text", "content": "增量文本片段" }
```

前端按到达顺序拼接。

### `done` — 完成

```json
{
  "type": "done",
  "answer": "完整回答",
  "hits": [ /* Hit[] */ ],
  "context": "...",
  "session_id": "75b9800f61c940ed",
  "latency_s": 26.04,
  "usage": null,
  "session_meta": {},
  "needs_clarify": false,
  "needs_reuse": false,
  "retry_count": 0,
  "correlation_id": "...",
  "research": { /* ResearchMeta，专家模式 */ }
}
```

- `session_id` 由后端注入，前端据此维持多轮上下文。
- 专家模式下携带 `research` 概要。

### `error` — 错误

```json
{ "type": "error", "message": "错误描述" }
```

## 12. 环境变量

后端启动可通过以下环境变量配置：

| 变量 | 默认 | 说明 |
|------|------|------|
| `CONFIG_PATH` | — | pipeline 配置文件路径（如 `local_api_config.yaml`） |
| `API_HOST` | `0.0.0.0` | 监听地址（`run_api.py`） |
| `API_PORT` | `8080` | 监听端口（`run_api.py`） |
| `API_KEYS` | 空 | 逗号分隔的 API key，空则不鉴权 |
| `CORS_ORIGINS` | `*` | 允许的跨域来源（逗号分隔） |
| `UPLOAD_DIR` | `<cwd>/uploads` | 上传文件与知识库工作目录根 |
| `SESSION_DIR` | `logs/sessions` | 会话持久化目录 |
| `SESSION_TTL` | `1800` | 会话过期秒数 |
| `TASK_MAX_WORKERS` | `2` | 异步任务最大并发数 |

### 启动方式

```bash
# 方式一：run_api.py（自动设置 INFO 日志级别）
CONFIG_PATH=local_api_config.yaml .venv-api/bin/python run_api.py

# 方式二：uvicorn 直接启动
CONFIG_PATH=local_api_config.yaml CORS_ORIGINS="*" \
  .venv-api/bin/python -m uvicorn pipeline.api.app:app --host 0.0.0.0 --port 8080
```

> 应用在 lifespan 中已显式将 `pipeline.*` logger 设为 INFO，因此无论用哪种方式启动，检索流程日志都会被 `SessionLogHandler` 采集。
